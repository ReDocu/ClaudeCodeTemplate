## 1. 개요

**1인 사용자가 다량의 프로젝트를 분류·운영하기 위한 폴더 기반 템플릿**이다.
회사의 업무 프로세스(멀티팀 협업 구조)를 1인 회사에 도입한 형태 — **팀 = 사업 영역(도메인) 분류, 역할 = 작업 모드(기획/개발/패키지), 루트 = 포트폴리오 리뷰**의 2축 모델로 운영한다.
하나의 메인 폴더 아래에 팀별 작업 공간, 역할별 분석 에이전트, 대시보드, 결과물 저장소를 배치하고,
슬래시 명령어로 인수인계(handover)·보고(report)·대시보드 갱신을 표준화한다.
같은 구조 그대로 다인 협업으로 확장할 수 있다 (역할 폴더에 실제 사람을 배정).

핵심 설계 원칙 세 가지:

1. **격리(Isolation)** — 하위 폴더에서 시작한 세션은 상위 폴더에 접근할 수 없다. 팀은 자기 공간만 본다. 1인 사용에서는 권한 경계라기보다 **세션 컨텍스트 오염 방지·사고 모드 분리 장치**다.
2. **상속(Inheritance)** — CLAUDE.md는 상위 폴더에서 하위로 자동 로드되므로, 거버넌스 규칙은 모든 세션에 상속된다. (데이터는 격리되고, 규칙은 상속된다.)
3. **문서가 곧 상태(Docs as State)** — 세션은 휘발되지만 handover·report·dashboard 문서가 프로젝트의 영속 상태를 담는다.

## 2. 배경과 문제

- Claude Code 세션은 컨텍스트가 휘발된다. 세션 간 연속성은 문서로만 확보할 수 있다.
- 1인이 여러 프로젝트를 병행하면 전환 비용이 크고, 방치된 프로젝트는 조용히 썩는다 — "지금 어디까지 됐는지"를 아는 주체가 없어진다.
- 기획·개발·배포를 한 사람이 한 세션에서 오가면 사고 모드가 섞여 어느 것도 완성되지 않는다.
- 분석 결과의 품질이 세션마다 들쭉날쭉하다 — 탐색·설명·검증·권고의 절차가 표준화되어 있지 않다.

## 3. 목표 / 비목표

### 목표
- 1인 사용자가 **팀=사업 영역, 역할=작업 모드**의 2축으로 다수 프로젝트를 병행 운영할 수 있다 — 루트 대시보드가 방치 영역(STALE)을 드러낸다.
- 새 팀원(또는 새 세션)이 handover 문서만 읽고 5분 안에 작업을 이어받을 수 있다.
- 총괄자는 `/report_root` 한 번으로 전체 프로젝트 현황 대시보드를 최신화할 수 있다.
- 탐색 → 설명 → 검증 → 권고의 4단계 분석 파이프라인을 명령어 하나로 실행할 수 있다.
- 모든 산출물이 표준 양식·표준 명명 규칙을 따른다.

### 비목표
- CI/CD, 외부 협업 도구(Jira, Slack 등) 연동은 이번 범위가 아니다.
- 실시간 동시 편집·충돌 해결은 다루지 않는다 (git에 위임).

## 4. 용어

| 용어 | 정의 |
|---|---|
| 총괄자 | 메인 폴더 루트의 CLAUDE.md가 규정하는 최상위 관리 역할 |
| 팀 관리자 | `00_Team/ProjectTeam_{팀명}/CLAUDE.md`가 규정하는 팀 단위 관리 역할 — 구현(Builder) 역할 겸임 |
| 역할 작업 공간 | 팀 폴더 내 하위 폴더 — 표준 `01_planner`(기획·디자인) / `02_developer`(프로그램 개발 과정) / `03_package`(배포·최종 선정). 각각 CLAUDE.md + `handover_{역할}.md` + `.claude/settings.json`(공유 폴더 접근)을 가진다 |
| 공유 폴더 | 역할 세션 전원이 접근하는 팀 내 폴더 — `00_Project`(작업물) / `11_team_doc`(문서 취합) / `90_result_output`(완료 백업, 읽기). 타 역할 폴더·`10_Dashboard`는 비공유 |
| 프로젝트 | 기획→개발→패키지를 관통하는 작업 단위. `00_Project/NN_{프로젝트명}/`에서 팀 직속으로 관리 (독립 git 저장소 연결 가능) |
| process.md | 프로젝트의 **유일한 상태 문서** — 목표/완료 조건·현재 단계·진행 기록·다음 할 일·블로커. **갱신은 팀 관리자 전담** (팀원 handover의 `[NN_프로젝트명]` 태그를 취합해 반영). 대시보드 파싱 소스를 겸한다 |
| 팀 구조 양식 | `00_Team/_ProjectTeam_Template/` — `/new_team`이 복사하는 팀 폴더 구조의 원본 (언더스코어 접두 = 팀 스캔에서 자연 제외) |
| handover | 다음 세션/담당자를 위한 인수인계 문서. **컨텍스트를 비웠을 때 다시 읽어 맥락을 복원하는 용도** — 작업 시작 시 읽고, 종료 시 갱신한다 |
| report | 대시보드 갱신을 위한 정기 현황 보고 문서 |
| 결과물 | `11_doc_result`에 저장되는 최종 산출 문서 |
| 파이프라인 | Explorer→Educator→Critic→Advisor 4단계 분석 절차 |

## 6. 구성 요소 상세

### 6.1 루트 CLAUDE.md (총괄자)
- 전체 폴더 구조와 각 폴더의 용도 명시.
- 문서 표준(8장)과 운영 규칙(9장)의 요약본 포함.
- 하위 폴더 세션에도 자동 상속되므로, **모든 세션이 지켜야 할 규칙만** 담는다 (루트 전용 지시는 별도 구분).

### 6.2 00_Team — 팀 작업 공간
- 팀 폴더 명명: `ProjectTeam_{팀명}` (팀명은 **영문만** — 한글은 특이 경우에 한해 최대한 짧게).
- 팀 구조 양식: `00_Team/_ProjectTeam_Template/`이 원본. `/new_team`이 이를 복사하고 예시 프로젝트(`01_Project01`)를 삭제한 뒤 `{팀명}`·`{YYYY-MM-DD}`를 치환한다.
- **필수 구성** (모든 팀 공통 — 삭제·개명 금지):

| 구성 | 성격 | 용도 |
|---|---|---|
| `00_Project/` | 공유 | **진행 중 프로젝트 작업물** — `NN_{프로젝트명}/` (독립 git 저장소 가능) + `process.md` |
| `10_Dashboard/` | 관리 | 팀 현황판 `DASHBOARD.md` — `/report`가 갱신, 루트가 수거 |
| `11_team_doc/` | 공유 | **문서 취합** — 회의록·공유 문서·팀 보고서(`report_{날짜}.md`)·파이프라인 결과 |
| `90_result_output/` | 보관 | **완료 후 백업** — 패키지 역할이 최종 선정한 작업물 (90번대 = 보관) |
| `CLAUDE.md` | 관리 | 팀 관리자 정의 (Builder 겸임 · process.md 전담) |

- **접근 권한 매트릭스 (팀 내 상호 호환)** — 역할 세션은 자기 역할 폴더에서 시작하되, 각 역할 폴더의 `.claude/settings.json`(additionalDirectories)이 공유 폴더 접근을 허용한다:

| 폴더 | planner | developer | package | 팀 관리자 |
|---|---|---|---|---|
| `00_Project/` 작업물 | ✅ | ✅ (코드) | ✅ (검증) | ✅ |
| `00_Project/*/process.md` | 읽기 | 읽기 | 읽기 | ✅ **전담 갱신** |
| `11_team_doc/` | ✅ | ✅ | ✅ | ✅ |
| `90_result_output/` | 읽기 | 읽기 | ✅ **쓰기 주체** | ✅ |
| 타 역할 폴더 / `10_Dashboard` | ❌ | ❌ | ❌ | ✅ |

- **프로젝트 관리 (00_Project)**:
  - 생성: 팀 폴더에서 `/new_project` → `00_Project/NN_{프로젝트명}/process.md` (번호 = 생성순 고정).
  - `process.md`가 프로젝트의 **유일한 상태 문서** (CLAUDE.md·handover.md 없음). **갱신은 팀 관리자 전담** — 팀원은 `handover_{역할}.md`에 `[NN_프로젝트명]` 태그로 기록하고, 팀 관리자가 팀 `/handover`로 취합하며 반영한다. 수명주기(단계 전환·우선순위·목표일·`[보류]`/`[종료]`)도 팀 관리자 전담이며 변경 시 진행 기록에 `[팀 관리자]` 표기.
  - **프로젝트 폴더 직접 세션은 없다** — 팀원은 역할 세션의 공유 접근으로, 구현은 팀 관리자(Builder 겸임)가 수행.
  - git 연결: 프로젝트 폴더는 독립 git 저장소일 수 있고, 템플릿 저장소는 process.md만 추적한다 (루트 .gitignore).
  - 종료: 패키지가 `90_result_output/` 백업 → `[종료]` 표기 → 아카이브 후보.

- 역할 작업 공간 (표준 양식 3개, 각각 CLAUDE.md + `handover_{역할}.md` + `.claude/settings.json`):

| 폴더 | 역할 | 임무 |
|---|---|---|
| `01_planner/` | 기획 | **기획·디자인 담당** — 요구사항·기획서·디자인 문서 (산출물은 00_Project에) |
| `02_developer/` | 개발 | **프로그램 개발 과정 담당** — 코드 작업은 `00_Project/`에서 수행 |
| `03_package/` | 패키지 | **배포·최종 선정 담당** — 검증·패키징, `90_result_output/` 백업 주체 |

- **커스텀 역할**: `NN_{역할}` 권장. **세션을 주는 폴더에는 `CLAUDE.md` + `handover_{역할}.md` 필수**, 공유 접근이 필요하면 settings.json 복사.
- 작업 흐름: 기획 → 개발 → 패키지. 역할 간 전달은 `handover_{역할}.md`(프로젝트 태그 포함), 프로젝트 진행은 팀 관리자가 `process.md`로 통합 기록.
- **격리**: 역할 세션은 자기 역할 폴더 + 공유 3폴더만 — 타 역할·타 팀·루트는 보지 못한다.

### 6.3 01~04 역할 폴더
각 역할은 두 가지로 구현된다:
- **폴더**: 역할 정의(CLAUDE.md)와 산출물(output/) 보관소.
- **서브에이전트 정의**(`.claude/agents/*.md`): `/pipeline` 명령이 호출하는 실행 주체.

| 역할 | 임무 | 권한 |
|---|---|---|
| 01_Explorer | 코드베이스 탐색, 구조·의존성·데이터 흐름 파악 | 읽기 전용. 코드 수정 금지 |
| 02_Educator | Explorer의 발견을 정확하면서 이해하기 쉽게 문서화 | 문서 작성만 |
| 03_Critic | 결론·설계·코드·문서의 허점·리스크·반례를 적대적으로 검증 | 읽기 + 검증 보고서만. 수정 제안은 하되 직접 수정 금지 |
| 04_Advisor | Critic 검증을 통과한 내용 기반으로 명확한 권고 (선택지 나열이 아닌 단일 권고 + 근거) | 문서 작성만 |

### 6.4 10_Dashboard — 2계층 대시보드
대시보드는 2계층으로 운영한다:
- **루트 `10_Dashboard/DASHBOARD.md`** — 전체 현황판. `/report_root`가 전 팀 일괄 작업 후 갱신 (진실의 원천).
- **팀 `10_Dashboard/DASHBOARD.md`** — 팀 현황판 (상태 요약·역할별 현황·블로커·최근 결과물). 팀 `/report`가 갱신하고 루트가 수거한다.
- **웹페이지 통합 (예정)**: 각 팀 대시보드를 연결한 웹페이지(HTML)를 템플릿 완성 후 제작한다. md 파일이 진실의 원천이고 HTML은 열람용 뷰다.

루트 `DASHBOARD.md` 표준 양식:

```markdown
# 프로젝트 현황판          (최종 갱신: YYYY-MM-DD, /report_root)

## 팀 현황
| 팀 | 프로젝트 | 상태 | 진행률 | 최근 보고 | 다음 마일스톤 | 링크 |
|---|---|---|---|---|---|---|
| TeamA | 결제모듈 | 🟢 정상 | 70% | 2026-07-06 | 7/15 베타 | [handover](...) |
| TeamB | 로그개선 | 🟡 지연 | 40% | 2026-06-28 ⚠STALE | 미정 | [handover](...) |

## 크로스팀 이슈 / 블로커
- TeamB가 TeamA의 API 스펙 확정을 대기 중 (since 2026-06-25)

## 최근 결과물 (최신 5건)
- 2026-07-06_TeamA_성능분석.md
```

- **신선도 규칙**: 최근 보고일이 7일을 초과하면 `⚠STALE` 자동 표기.
- 상태등: 🟢 정상 / 🟡 지연·주의 / 🔴 블로킹 / ⚪ 대기·중단.

### 6.5 11_doc_result
- 파일명: `YYYY-MM-DD_{팀명}_{제목}.md` (날짜는 ISO 형식 고정).
- `INDEX.md`: 날짜 역순 표(날짜/팀/제목/유형/링크). `/report_root`와 `/pipeline`이 자동 갱신.

### 6.6 90_Templates [추가 제안]
- 문서 양식(handover·report·result)의 원본. 명령어들은 문서 생성 시 반드시 이 템플릿에서 출발한다.
- 팀 폴더 구조의 원본은 `00_Team/_ProjectTeam_Template/`이다 — `/new_team`이 이를 복사·치환하여 스캐폴딩한다. `90_Templates`의 CLAUDE 템플릿은 양식 폴더 유실 시의 예비 사본으로 함께 갱신한다.
- `/new_project`는 명령어 파일에 내장된 양식으로 스캐폴딩한다 (역할 세션은 `90_Templates` 접근 불가).

### 6.7 99_Archive [추가 제안]
- 종료된 프로젝트 폴더는 `99_Archive/teams/{팀명}/{프로젝트명}/`으로 이동.
- 90일 경과한 결과물은 `99_Archive/doc_result/YYYY/`로 이동 (`/report_root` 실행 시 이동 대상을 제안하고, 사용자 확인 후 이동).
- 대시보드·INDEX에서 아카이브 항목은 접힌 섹션으로 분리.

### 6.8 manual
- `00_quickstart.md`가 진입점: "어느 폴더에서 세션을 열지 → 무슨 명령을 칠지"를 시나리오별로 안내.
- **사용자 맥락별 매뉴얼 3종**: `05_root_manager.md`(root 관리자) / `06_team_manager.md`(팀별 관리자) / `07_individual_manager.md`(개별 관리자 — 역할 담당자). 각 매뉴얼은 "누구인가 → 어디서 세션을 여는가 → 무엇을 관리하는가 → 명령·루틴 → 금지 사항" 구조.
- 명령어 레퍼런스는 7장 사양과 항상 동기화.

### 6.9 handover_root.md
- 각 팀의 `handover.md`(팀 인수인계)를 취합·요약한 전체 인수인계 문서.
- 원문 복사가 아니라 **요약 + 원문 링크** 구조 (문서 비대화 방지).

## 7. 명령어 사양

공통 규칙:
- 모든 명령어는 실행 전 **현재 위치를 판별**하고, 허용되지 않는 위치에서는 이유와 함께 거부한다.
- **root 명령 원칙**: `*_root` 명령은 전 팀 **일괄 작업**을 먼저 수행한 뒤, 결과를 최종 관리자(템플릿 최상위 폴더)의 문서(대시보드·색인·handover_root)로 취합해 루트 기준으로 관리한다.

## 9. 운영 규칙

### 9.1 접근 범위 (격리 + 팀 내 공유)
- 하위 폴더 세션은 상위 폴더 파일에 접근하지 않는다 (Claude Code 기본 권한 동작에 의해 강제됨).
- **팀 내 공유 예외**: 역할 세션은 자기 팀의 공유 폴더(`00_Project`/`11_team_doc`/`90_result_output`)에 접근할 수 있다 — 각 역할 폴더의 `.claude/settings.json`(`permissions.additionalDirectories`)이 허용한다. 미지원 환경에서는 `claude --add-dir` 또는 프롬프트 승인으로 동일 범위를 허용한다 (매트릭스가 규칙).
- 타 역할 폴더·`10_Dashboard`·타 팀·루트는 여전히 금지.
- 단, 상위 CLAUDE.md의 **규칙은 자동 상속**된다 — 각 CLAUDE.md는 이를 전제로 작성한다.
- 루트 세션은 모든 하위 폴더에 접근 가능하다 (하향 접근은 허용).

### 9.2 역할 권한
- Explorer·Critic은 코드를 **수정하지 않는다**. Educator·Advisor는 문서만 생성한다.
- 실제 코드 수정은 `00_Project/{프로젝트}/` 안에서만 — 수행 주체는 **개발 역할 세션(공유 접근)** 또는 **Builder를 겸임하는 팀 관리자**.
- `process.md` 갱신은 **팀 관리자 전담** — 팀원은 handover에 태그로 기록만 한다.
- 기획(`01_planner`)은 기획·디자인 문서, 패키지(`03_package`)는 검증·패키징·`90_result_output` 백업 — 소스 결함은 handover 태그로 개발에 되돌린다.

### 9.3 갱신 주기 (권장)
- `/handover`: 작업 세션 종료 시마다.
- `/report_root`: 주 1회 이상. 7일 초과 미보고 팀은 대시보드에 STALE 표기.

### 9.4 아카이브
- 종료 프로젝트·90일 경과 결과물은 `99_Archive`로 이동 (사용자 확인 후).

### 9.5 Git
- 커밋 단위: 문서 갱신 명령 1회 = 커밋 1회.
- 커밋 메시지: `docs(handover): TeamA 결제모듈`, `docs(dashboard): 주간 갱신 2026-07-06` 형식.

| 4역할 | `.claude/agents/{explorer,educator,critic,advisor}.md` — 페르소나·권한(도구 제한)·출력 양식 정의. Explorer/Critic은 읽기 전용 도구만 허용 |
| `/report_root`의 팀 병렬 보고 | 루트 세션이 팀별 서브에이전트를 병렬 실행 (하향 접근 허용이므로 가능) |
| 격리 | Claude Code 기본 권한(작업 디렉터리 밖 읽기 제한) + 각 CLAUDE.md에 명시적 금지 조항 |
| 권한 설정 | `.claude/settings.json` — 문서 생성·읽기 도구 allowlist로 프롬프트 최소화 |

### 구현 순서 (마일스톤)
1. **M1 — 뼈대**: 폴더 구조 + 루트 CLAUDE.md + 90_Templates + manual 초안
2. **M2 — 인수인계**: `/handover`, `/handover_root` + handover 템플릿 검증
3. **M3 — 보고·대시보드**: `/report`, `/report_root`, `/status` + DASHBOARD/INDEX 자동 갱신
4. **M4 — 파이프라인**: 4역할 에이전트 + `/pipeline` + 반려 재순환
5. **M5 — 스캐폴딩·아카이브**: `/new_team`, `/new_project`, 아카이브 규칙
6. **M6 — 실전 검증**: 실제 프로젝트 1개를 템플릿 위에서 2주 운영 후 manual/양식 보완

## 11. 성공 지표

- 새 세션이 handover만 읽고 작업 재개까지 걸리는 시간 ≤ 5분.
- `/report_root` 1회 실행으로 대시보드가 사람 손 없이 최신화됨.
- 대시보드 STALE 팀 비율 < 20% 유지.
- 결과물 문서의 명명 규칙 준수율 100% (명령어가 생성하므로 구조적으로 보장).

## 12. 오픈 이슈 (v1.2 갱신 반영)

1. **Builder(구현 역할)** — ✅ 갱신(2026-07-07): 별도 Builder 에이전트는 추가하지 않되, **구현(Builder) 역할은 팀 관리자가 겸임**한다. 구현 작업 공간은 `02_developer/`. 분석 4역할은 분석 전용을 유지한다 (Critic 검증의 독립성 확보).
2. **팀명 한글 허용 여부** — ✅ 갱신(2026-07-07): **영문만 사용**. 한글은 특이 경우에 한해 허용하되 최대한 짧게. `/new_team`이 한글 입력 시 리스크 안내 후 사용자 확인을 받고 진행.
3. **dashboard.html 제공 여부** — 🔜 갱신(2026-07-07): 템플릿 완성 후 작업 예정. **각 팀 `10_Dashboard/`를 연결한 웹페이지**로 제작한다. md 파일이 진실의 원천이고 HTML은 열람용 뷰. 상세 사양: **[PRD_dashboard.md](PRD_dashboard.md)** (관리자/팀 2계층, 폴더 스캔 + handover 파싱 기반 갱신, `/dashboard` 명령).
4. **명령어 인식 범위 검증** — ⏳ M6에서 실측. 단일 소스(루트 `.claude/commands`)로 시작하되, 하위 세션 격리에 대비해 **모든 명령어에 필요 양식을 내장**했고, 미인식 시 대응(복제 배포/사용자 레벨 설치)을 `manual/04_faq.md`에 문서화.
5. **/report와 /report_root의 중복** — ✅ 결정: 둘 다 유지. `/report_root`는 팀 `11_team_doc/`의 7일 이내 `report_*.md`가 있으면 재사용하므로 중복 작업이 발생하지 않음.
6. **root 명령 관리 원칙** — ✅ 갱신(2026-07-07): `*_root` 명령은 전 팀 **일괄 작업 이후, 최종 관리자(템플릿 최상위 폴더) 기준으로 취합·관리**한다 (7장 공통 규칙에 반영).
