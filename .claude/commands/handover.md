description: 인수인계 갱신 — 역할 폴더는 handover_{역할}, 팀 폴더는 process.md 갱신 + 취합
---

현재 폴더 기준으로 상태 문서를 갱신한다. 아래 절차를 따르라.

## 1. 위치 판별
`pwd`로 현재 경로를 확인한다.
- **역할폴더** = 팀 폴더 바로 아래 폴더 중 필수 폴더(`00_Project`/`10_Dashboard`/`11_team_doc`/`90_result_output`)를 제외한 것. 표준: `01_planner`/`02_developer`/`03_package`, 커스텀 역할도 가능 (`CLAUDE.md` + `handover_{역할}.md` 보유).
- 경로가 `.../00_Team/{팀폴더}/{역할폴더}`로 끝남 → **역할 모드**
- 경로가 `.../00_Team/{팀폴더}`로 끝남 → **팀 모드** (팀 관리자)
- 경로가 `.../00_Project/{프로젝트}` → 프로젝트 직접 세션은 폐지됨. 팀 폴더에서 실행하라고 안내하고 **중단**. (process.md 갱신은 팀 관리자 전담)
- 현재 폴더에 `00_Team/`과 `manual/`이 존재 → 루트다. `/handover_root`를 안내하고 **중단**.
- 어느 것도 아니면 사용자에게 위치를 확인하고 중단.

## 2. 근거 수집
- 기존 상태 문서와 현재 폴더의 CLAUDE.md를 읽는다.
- `git log --oneline -20 -- .`으로 최근 변경을 확인한다.
- 이번 세션에서 수행한 작업 내용을 근거로 삼는다. **추측으로 채우지 않는다** — 확인 안 되는 항목은 "(미확인)".

## 3-A. 역할 모드 → `./handover_{역할}.md` 갱신 (덮어쓰기)

표준 양식 (원본: 90_Templates/handover.template.md — 하위 세션 접근 불가로 여기 내장):

```markdown
# Handover — {대상명} (갱신: {오늘 날짜 YYYY-MM-DD})

## 목표
(이 역할이 팀 목표에 기여하는 것 — 1~2문장)

## 현재 상태
(한 문단 요약 — 새로 온 사람이 이것만 읽어도 감이 잡히게)

## 완료된 것
- [NN_프로젝트명] (완료 항목 — 결과물·파일 경로 포함. 프로젝트 무관 항목은 태그 생략)

## 진행 중인 것
- [NN_프로젝트명] (항목 — 마지막 작업 시점, 어디까지 됐는지)

## 다음 할 일 (우선순위 순)
1. (가장 급한 것)

## 주의사항 / 함정
- (다음 사람이 반드시 알아야 할 것 — 이유 포함)

## 핵심 파일 경로
- `경로` — 역할 한 줄
```

**프로젝트 관련 항목에는 반드시 `[NN_프로젝트명]` 태그**를 붙인다 — 팀 관리자가 이 태그로 process.md에 반영한다.

**태그 누락 검사 (문서를 쓰기 전 필수)**: 이번 세션에서 `../00_Project/NN_{프로젝트명}/` 하위 파일을 읽거나 수정했다면, 그 프로젝트의 태그가 붙은 항목이 최소 1개 있어야 한다. 없으면 어느 항목에 태그를 붙일지 제안하고 사용자 확인 후 진행한다. **1인 사용자도 생략 금지** — 취합 시점(관리자 모드)의 나는 이 태그만 보고 분류한다.

## 3-B. 팀 모드 (팀 관리자) — 두 단계를 순서대로 수행

### ① 각 프로젝트 process.md 갱신
1. 각 역할 폴더의 `handover_{역할}.md`를 모두 읽는다.
2. `[NN_프로젝트명]` 태그가 붙은 항목을 프로젝트별로 모아, 해당 `00_Project/NN_{프로젝트명}/process.md`에 반영한다:
   - 「진행 기록」에 날짜 역순으로 추가 (출처 역할 표기 — 예: `2026-07-08: (개발) 재시도 로직 구현, 테스트 12/16`)
   - 「상태 요약」의 상태등·진행률을 근거에 맞게 조정, 「다음 할 일」·「블로커」 갱신
   - 제목의 갱신일을 오늘로
3. 수명주기 판단(팀 관리자 전담): 단계 완료 근거가 있으면 「현재 단계」 굵게 이동, 필요 시 우선순위·목표일·`[보류]`/`[종료]` 표기 — **변경 시 진행 기록에 `[팀 관리자]` 한 줄**을 남긴다.
4. **태그 누락 의심 감지**: 태그는 없지만 프로젝트성으로 보이는 항목(프로젝트명·`00_Project` 경로·산출물 언급 등)을 「태그 누락 의심」 목록으로 사용자에게 보여주고, 확인된 항목은 태그를 붙여 2와 같이 반영한다.
5. 확인되지 않았거나 매칭 프로젝트가 없는 항목은 **버리지 않는다** — ②의 「미분류 (보류함)」 섹션에 요약과 출처(역할·날짜)를 남긴다. **유실 방지의 실체는 이 보류함이다** (태그 감지는 완벽할 수 없지만, 감지에 실패해도 기록은 남는다).

### ② 팀 handover.md 취합 (덮어쓰기)
위 handover 양식에 두 섹션을 추가해 `./handover.md`를 작성한다 (원문 통째 복사 금지, 요약 + 상대경로 링크):
- `## 프로젝트별 요약 (00_Project)` — 프로젝트마다 단계/상태 한 줄 + `[process](./00_Project/{프로젝트}/process.md)` 링크
- `## 역할별 요약` — 역할마다 3줄 요약 + `[handover](./{역할폴더}/handover_{역할}.md)` 링크
- `## 미분류 (보류함)` — ①-5의 항목들 (출처 포함). 이후 취합에서 분류되면 제거한다. 항목이 없으면 섹션 생략.
문서가 없으면 "미작성"으로 표기.

## 4. 마무리
- 갱신된 문서의 3줄 요약을 사용자에게 보여준다. (팀 모드: 갱신된 process.md 목록 + 태그 누락 의심·미분류 잔여 건수 포함)
- 커밋을 제안한다: `docs(handover): {대상명}`
