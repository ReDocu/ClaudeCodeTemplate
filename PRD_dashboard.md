## 1. 개요

md 대시보드(진실의 원천)를 사람이 한눈에 보기 좋은 **HTML 웹페이지**로 자동 생성한다.
대시보드는 2계층이며 서로 링크로 연결된다:

| 계층 | 이름 | 목적 | 위치 |
|---|---|---|---|
| 루트 | **관리자 대시보드** | 팀들의 현황 파악 | `10_Dashboard/dashboard.html` |
| 팀 | **팀 대시보드** | 팀원(역할)들의 현황 파악 | `00_Team/ProjectTeam_{팀명}/10_Dashboard/dashboard.html` |

핵심 원칙:
1. **md가 진실의 원천, HTML은 열람용 뷰** — HTML을 손으로 고치지 않는다. 고칠 내용은 handover/report에 쓴다.
2. **현황은 두 소스에서 갱신된다**: ① 진행 중인 프로젝트(`00_Project/`) **폴더 구조 스캔** ② **handover·process 파일 파싱**. 둘 다 컨텍스트 복원용 문서이므로 항상 최신 상태라는 전제를 활용한다.
3. **root 명령 원칙**: 전 팀 일괄 스캔 → 최종 관리자(루트) 기준으로 일괄 생성·관리.

## 2. 배경과 문제

- md 대시보드는 git 친화적이지만, 여러 팀·여러 역할의 현황을 한눈에 훑고 클릭으로 오가기 어렵다.
- 이 템플릿에서 "현재 상태"는 handover 파일과 폴더 구조에 이미 존재한다 — 별도 입력 없이 **읽어서 시각화만 하면 된다**.
- 관리자는 "어느 팀이 막혀 있는가"를, 팀 관리자는 "어느 역할이 막혀 있는가"를 5초 안에 파악해야 한다.

## 3. 목표 / 비목표

### 목표
- 루트에서 명령 1회로 **관리자 + 전 팀 대시보드 HTML을 일괄 재생성**한다.
- 브라우저에서 `file://`로 열어 관리자 ↔ 팀 페이지를 상대경로 링크로 왕복할 수 있다.
- handover 갱신일 기준 신선도(`⚠STALE`)와 상태등(🟢🟡🔴⚪)을 시각적으로 표시한다.
- handover 파싱에 실패한 팀/역할이 있어도 전체 생성은 중단되지 않는다 (⚠ 표기).

### 비목표
- 서버·DB·실시간 동기화·인증·외부 배포(호스팅) — 정적 파일만 다룬다.
- HTML에서의 편집 기능 — 열람 전용.
- md 대시보드의 대체 — `DASHBOARD.md`는 계속 진실의 원천으로 유지된다.

## 4. 사용자와 화면

### 4.1 관리자 대시보드 — 팀들의 현황 파악

- **사용자**: root 관리자 (총괄자).
- **화면 구성**:

```
┌─────────────────────────────────────────────────────┐
│ 프로젝트 현황판                생성: 2026-07-07 14:00 │
│ 팀 4 · 🟢2 🟡1 🔴1 · STALE 1                        │
├─────────────────────────────────────────────────────┤
│ [팀 카드 그리드 — 카드 클릭 시 팀 대시보드로 이동]     │
│ ┌───────────────┐ ┌───────────────┐                 │
│ │ 🟢 TeamA  70%  │ │ 🔴 TeamB  40% │                 │
│ │ 마일스톤: 7/15 │ │ ⚠STALE 9일    │                 │
│ │ 블로커: 없음   │ │ 블로커: API   │                 │
│ └───────────────┘ └───────────────┘                 │
├─────────────────────────────────────────────────────┤
│ 크로스팀 이슈 / 블로커 (시작일 포함)                  │
│ 최근 결과물 (11_doc_result 최신 5건)                 │
│ 아카이브 후보                                        │
└─────────────────────────────────────────────────────┘
```

- **팀 카드 데이터**: 팀명 / 상태등 / 전체 진행률 / 다음 마일스톤 / 최근 갱신일(+STALE) / 블로커 유무 / 역할 상태 요약(🟢🟢⚪) / 팀 대시보드 링크.

### 4.2 팀 대시보드 — 팀원(역할)들의 현황 파악

- **사용자**: 팀 관리자, 역할 담당자(개별 관리자).
- **화면 구성**:

```
┌─────────────────────────────────────────────────────┐
│ 팀 현황판 — TeamA          생성: … · [← 관리자 대시보드] │
│ 🟢 70% · 마일스톤 7/15                               │
├─────────────────────────────────────────────────────┤
│ [역할 카드 — 역할별 현황]                             │
│ ┌─ 🟢 기획 (01_planner) ── 갱신 07-06 ─┐             │
│ │ 현재 상태: … (handover 요약)          │             │
│ │ 다음 할 일: 1. … 2. … 3. …           │             │
│ └───────────────────────────────────────┘             │
│ (developer / package / 커스텀 역할 동일)              │
├─────────────────────────────────────────────────────┤
│ 진행 중 프로젝트 (폴더 스캔)                          │
│ │ 00_Project/01_my-feature  개발 단계  🟢  갱신 07-05 │
│ 블로커 / 역할 간 대기                                 │
│ 최근 백업 결과물 (90_result_output 최신 5건)          │
└─────────────────────────────────────────────────────┘
```

- **역할 카드 데이터**: 역할명·폴더 / 상태등 / `handover_{역할}.md`의 「현재 상태」 요약 / 「다음 할 일」 상위 3건 / 갱신일(+STALE).
- **프로젝트 행 데이터** (`00_Project` 스캔 + `process.md` 파싱): 프로젝트명 / 현재 단계(기획/개발/패키지) / 상태등 / 진행률 / 갱신일 / 블로커.

## 5. 데이터 소스와 갱신 규칙

현황은 다음 두 경로로 갱신된다 (둘 다 **읽기 전용 스캔** — 생성기는 md를 수정하지 않는다):

### 5.1 진행 중인 프로젝트 폴더 기반 (구조 스캔)

| 대상 | 스캔 규칙 |
|---|---|
| 팀 목록 | `00_Team/ProjectTeam_*` 열거. **양식 폴더(`ProjectTeam_양식[팀명]`) 제외** |
| 역할 목록 | 팀 폴더 바로 아래 폴더 중 필수 폴더(`00_Project`/`10_Dashboard`/`11_team_doc`/`90_result_output`) 제외. `CLAUDE.md`+`handover_*.md` 보유 여부로 유효성 판정 (없으면 ⚠ 미구성 표기) |
| 진행 중 프로젝트 | `00_Project/NN_*` 폴더 열거 — `process.md` 보유 여부로 유효성 판정. process.md 제목에 `[종료]` 표기 시 "종료"로 분류 (아카이브 후보) |
| 최근 결과물 | 팀 `90_result_output/`(완료 백업)·`11_team_doc/`(보고서·문서), 루트 `11_doc_result/` — 파일명 날짜 역순 5건 |

### 5.2 handover / process 파일 기반 (내용 파싱)

handover와 process.md는 컨텍스트 복원용 문서 — **작업 종료 시마다 갱신되므로 최신 현황의 1차 소스**다.

역할 `handover_{역할}.md` 파싱:

| 추출 항목 | 파싱 규칙 (표준 양식 기준) |
|---|---|
| 갱신일 | 제목 행의 `(갱신: YYYY-MM-DD)` |
| 현재 상태 | `## 현재 상태` 섹션 첫 문단 (카드 요약에 사용, 최대 2줄로 절삭) |
| 다음 할 일 | `## 다음 할 일` 목록 상위 3건 |
| 진행 중인 것 | `## 진행 중인 것` 목록 |
| 블로커 | `## 주의사항 / 함정` + report의 블로커 섹션 (있다면) |
| 상태등·진행률·마일스톤 | 팀 `10_Dashboard/DASHBOARD.md`·최신 `report_*.md`의 상태 요약 표 (있다면). 없으면 handover 기반 추정 없이 "미보고" 표기 |

프로젝트 `process.md` 파싱 (프로젝트 카드/행의 소스):

| 추출 항목 | 파싱 규칙 (표준 양식 기준) |
|---|---|
| 갱신일 | 제목 행의 `(갱신: YYYY-MM-DD)` · 제목 앞 `[종료]` → 종료 분류 |
| 현재 단계 | `## 현재 단계`의 굵게 표시된 단계 (기획/개발/패키지) |
| 상태등·진행률·담당·목표일 | `## 상태 요약` 표 |
| 최근 진행 | `## 진행 기록` 최상단 1~2건 |
| 다음 할 일 | `## 다음 할 일` 상위 3건 |
| 블로커 | `## 블로커` 섹션 ("없음"이 아니면 표시) |

- **신선도**: 갱신일이 오늘 기준 **7일 초과 → `⚠STALE`**. 갱신일 파싱 실패 시 git 최종 커밋일로 대체하고 `(git 추정)` 표기.
- **휴면**: 팀 `handover.md`의 「현재 상태」에 `휴면` 선언(기한부: `휴면 ~YYYY-MM-DD`)이 있으면 STALE 대신 **휴면 배지** 표시 — 의도된 멈춤과 방치를 구분한다 (1인 다프로젝트 운영에서 휴면 팀은 상시 존재).
- **파싱 실패 내성**: 비표준 양식·섹션 누락은 해당 항목만 "(미확인)"으로 남기고 계속 진행. 파일 자체가 없으면 카드에 "handover 미작성" 경고.

## 6. 생성 방식

| 항목 | 내용 |
|---|---|
| 생성 방식 | **셸 + DATA 블록** — 루트 `10_Dashboard/dashboard.html`이 셸(마크업·CSS·JS)의 단일 원본. 렌더링은 셸 내 JS가 수행하고, 데이터는 `DASHBOARD-DATA-START/END` 마커 사이 `const DATA`·`const CFG` 두 줄에 내장. 명령(Claude 세션)이 스캔·파싱 후 **이 두 줄만 교체**한다 — 파이썬 등 외부 도구 불필요, 셸 불변으로 시각적 결정성 보장. (구 파이썬 생성기는 `99_Archive/2026-07-07_dashboard_generator/`로 은퇴) |
| 실행 위치 | **루트 전용** (root 명령 원칙 — 전 팀 일괄 스캔 후 루트 기준 생성) |
| 출력 | 루트 `10_Dashboard/dashboard.html`(DATA 갱신) + 각 팀 `10_Dashboard/dashboard.html`(루트 셸 사본 + 자기 팀 데이터만 — 격리) |
| HTML 형식 | **단일 파일** — CSS/JS 인라인, 외부 CDN·폰트·이미지 요청 금지 (`file://`·오프라인 동작 보장) |
| 링크 | 상대경로만 사용: 관리자 → `../00_Team/ProjectTeam_{팀명}/10_Dashboard/dashboard.html`, 팀 → 역방향 |
| 데이터 | 생성 시점에 HTML에 내장 (열람 시 재수집 없음). 페이지 상단에 생성 시각 표기 |
| 인코딩 | UTF-8 (한글 팀명 특이 경우 대비) |

## 7. 명령어 사양 — `/dashboard` [신설]

| 항목 | 내용 |
|---|---|
| 실행 위치 | 루트 전용 (아니면 이유와 함께 거부) |
| 동작 | ① 전 팀 스캔 + handover/process 파싱 ② 루트 dashboard.html의 DATA 블록 교체 ③ 팀 페이지 갱신 (셸 사본 + 팀 데이터) ④ 요약 출력 (팀 수, 상태 분포, STALE/휴면, 파싱 실패 목록) — DATA 스키마는 `.claude/commands/dashboard.md`에 내장 |
| 연동 | `/report_root` 6단계·`/handover_root` 4단계에서 동일 절차 수행 (md 갱신 → HTML 갱신 순서) |
| 산출물 | `dashboard.html` (루트 1 + 팀 N) |
| 커밋 제안 | `docs(dashboard): HTML 재생성 {YYYY-MM-DD}` |

갱신 주기 권장: `/report_root` 실행 시 자동 + 필요 시 수시 (`/dashboard` 단독 실행은 md를 건드리지 않으므로 부담 없음).

## 8. 수용 기준 (Acceptance Criteria)

1. 루트에서 `/dashboard` 1회 실행으로 관리자 + 전 팀 HTML이 갱신된다 — **파이썬 등 외부 도구가 설치되지 않은 환경에서도 동작한다.**
2. `file://`로 관리자 대시보드를 열어 팀 카드 클릭 → 팀 대시보드 → "관리자 대시보드" 링크로 복귀가 동작한다.
3. 역할 `handover_{역할}.md`를 갱신하고 재생성하면 해당 역할 카드의 현재 상태·다음 할 일·갱신일이 반영된다.
4. `00_Project/`에 새 프로젝트 폴더(`NN_{프로젝트명}/process.md`)를 만들고 재생성하면 "진행 중 프로젝트"에 단계·상태와 함께 나타난다.
5. handover가 없는 역할이 있어도 생성이 완료되고, 해당 카드에 "handover 미작성"이 표시된다.
6. 8일 전 갱신된 handover는 `⚠STALE`로 표시된다.
7. 외부 네트워크가 차단된 환경에서도 페이지가 깨지지 않는다 (외부 요청 0).
8. 팀 handover에 `휴면` 선언이 있으면 STALE 대신 휴면 배지가 표시되고, STALE 집계에서 제외된다.
9. 관리자 페이지의 「프로젝트 일람」에서 전 팀의 진행 중 프로젝트가 갱신일 역순으로 보이고, 행 클릭 시 소속 팀 뷰로 이동한다.
10. 대시보드 파일 상단·하단에 자동 생성 파일 경고(직접 수정 금지)가 표시된다.
