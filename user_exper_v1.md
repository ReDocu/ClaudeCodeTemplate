## 2. 발견사항 (심각도 순)

### F1 (높음) — wmux 브라우저 패널에서 페이지 타이머 스로틀 → 2.5s 폴링 무력화
- **증상**: 세션 kill 후 서버(`/api/state`)는 즉시 1/2로 갱신됐지만 UI는 15초+ 동안 "세션 2/2" 유지. 1초 카운터(rc++)조차 10초에 1회만 증가 — `setInterval` 전체가 ~1/분 수준으로 스로틀. `document.visibilityState`는 `visible`인데도 발생.
- **영향**: 이 제품의 자연스러운 사용처(wmux 패널에 대시보드 상주)에서 "실시간 관제"가 사실상 정지. 그런데 배지는 `● live`를 유지해 사용자가 스테일을 인지할 수 없음. 일반 크롬에서도 백그라운드 탭이면 동일 증상 예상(Chromium intensive throttling).
- **제안**: ① 마지막 성공 폴 시각을 `Date.now()` 차로 계산해 표시(타이머 증가 방식 금지 — F2와 동반 수정) ② 그 차가 임계(예 10s) 초과 시 `● live` 대신 `◐ stale` 경고 배지 ③ `visibilitychange`/포커스 복귀 시 즉시 loadState ④ 근본적으로는 폴링 대신 SSE 고려.

### F2 (중) — "⟳ N초 전" 신선도 표시가 폴링 성공과 무관
- **근거(소스 확정)**: `dashboard.html:958-960` — `rc`는 1초 interval로 증가만 하고, **리셋은 수동 [새로고침](doRefresh)에서만**. 자동 loadState 성공은 rc를 리셋하지 않음. 즉 "⟳ 6초 전"은 실제로는 "마지막 수동 새로고침/페이지 로드로부터 6초"라는 뜻 — 신선도 지표가 사실상 거짓.
- **제안**: loadState 성공 시점 타임스탬프 저장 + 표시 시 `Date.now()` 차 계산(F1 제안 ①과 동일 수정).

### F3 (중) — 드로어가 열려 있는 동안 내용이 갱신되지 않음
- **증상**: 드로어를 연 채 [▶ Claude 실행] → API는 `on` 전환 완료됐는데 드로어 헤더는 "❯터미널 (claude off)", 버튼은 "시작 중…"으로 무기한 잔류. 닫았다 다시 열어야 "✦claude 실행중" 반영.
- **영향**: 사용자가 가장 집중해서 보는 화면(방금 누른 버튼의 결과)이 가장 늦게 갱신됨. "실행이 실패했나?" 오해 → 재시도 유인.
- **제안**: 폴링 갱신 시 열린 드로어(DR)를 재렌더하거나, 최소한 doClaude 성공 콜백에서 드로어 헤더·버튼을 갱신.

### F4 (낮) — 역할 추가 다이얼로그에 입력 필드 2개 노출
- **증상**: [＋ 역할] 다이얼로그(namer 골격 재사용)에 "역할 이름"(nm-a)과 프로젝트 생성용 "역할 (쉼표 구분…)"(nm-b)이 **둘 다 visible**. 어느 칸에 입력해야 하는지 모호(실측: nm-a만 읽음).
- **제안**: 역할 추가 모드에서 nm-b 행 숨김.

### F5 (낮) — 기능 인벤토리에 `0.1.1` 플러그인 표기 + superpowers 중복
- **증상**: Global 인벤토리에 `superpowers PLUGIN`이 2회, 그리고 `0.1.1 PLUGIN`(버전 문자열이 이름 자리에) 표시. 드로어의 global 목록에도 동일 재현 — `caps.js`의 플러그인 스캔이 디렉터리/매니페스트 구조를 오파싱하는 것으로 추정.
- **제안**: 플러그인 이름 추출 로직 검증(이름 필드 우선, 버전 디렉터리 제외) + 이름 기준 dedupe.

### F6 (낮) — 포트맵 노이즈 (기존 Q6 그대로)
- **증상**: mysqld·steam×2·AsusSoftwareManager·OneDrive·Postman·delfino 등 시스템/앱 리스너가 `→ 기타`로 13건 나열 — 정작 관심 대상(프로젝트 귀속 포트)이 묻힘.
- **제안**: 기본 뷰는 "프로젝트 귀속 + dev 포트 대역"만, `기타 N개` 접기. `ports.js` NOISE 셋 보강.

### F7 (관찰) — usage 게이지: 학습 max 기반 "98%"의 오경보 소지 · week-opus 필터 실효성
- **증상**: 세션 중 max 학습 완료 후 `주간·전체 98%`(575M/589M) 표시 — 이 max는 플랜 실한도가 아니라 **관측 피크**라서, 실제로는 여유가 있어도 위기처럼 보임. 또한 `week-opus`가 `week`와 바이트 단위까지 동일 — 모델명 필터가 신형 모델(claude-fable-5 등)을 Opus로 분류하거나 필터가 무력한 정황.
- **제안**: `source:"history"`일 때 게이지에 "추정 한도" 워터마크(수치 병기 유지) · week-opus 모델 필터 목록을 실측 트랜스크립트 모델명으로 검증. 수동 키(`usageMax*`) 안내 노출(기존 Q5).

### F8 (관찰) — 비활성화 후 OS pwsh 1쌍 잔존 → 프로젝트 폴더 잠금 (계승 교훈 재현)
- **증상**: 비활성화가 wmux agent 3개는 정리했지만, 활성화 시각에 생성된 pwsh 1쌍(래퍼 + `-wd root/UxTest` 서버 프로세스)이 OS에 잔존 — 폴더 삭제 시 "Device or resource busy". role pane 셸들은 깨끗이 죽었고, **잔존한 것은 `-wd`가 프로젝트 루트를 가리키는 초기(workspace 생성 시) pane으로 추정**.
- **영향**: §7-A 테스트 가이드의 "탐색기에서 삭제" 단계가 실패 — 사용자는 원인 프로세스를 찾을 수 없음.
- **제안**: deactivate 후 pid 생존 확인 → 잔존 시 OS kill 폴백, 또는 최소한 로그/토스트로 "폴더 잠금 가능 — pwsh 확인" 안내.

## 3. 잘 작동한 것 (강점)

- **가드 체계가 전 구간 견고**: 살아있는 역할 409 · ops 400 · 보관 중 활성화 409 · claude 중복 기동 방지(on이면 버튼 자체 미노출) — 잘못된 조작이 애초에 어렵다.
- **멱등·수렴 계약이 실측에서 증명**: 복구는 죽은 세션만 재스폰(중복 0), 역할 재추가는 병합, 재활성화는 재사용 우선(`스폰 1 · 재사용 2` 로그).
- **비활성화 확인 다이얼로그**: 종료 대상 세션 목록 + "undo 없음" + 행동을 서술하는 버튼 라벨 — 파괴적 작업 UX의 모범.
- **관리 로그(KR4)**: 전 이벤트가 정확·가독성 있게 남음. 사후 추적 가능.
- **offline/demo 구분 배지**: 서버 단절과 파일 직접 열기를 다른 메시지로 안내.
- **pane kill 시 자식 프로세스까지 정리**: dev pane 종료 때 python 리스너(:5317)도 함께 소멸 — 계승 우려(자식 잔존) 대비 양호. (잔존 이슈는 F8의 초기 pane 1쌍으로 국한.)
- **다이얼로그 카피 품질**: "자유 스폰이 아니라 선언 경유 — 스폰은 활성화/[세션 복구]가 수행" 같은 문구가 시스템 모델을 정확히 가르쳐 줌.
