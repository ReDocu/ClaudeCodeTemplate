# main_manual — 처음 시작하는 분을 위한 안내서

> 이 문서 하나만 읽으면 템플릿을 바로 사용할 수 있습니다.
> 더 깊은 내용이 필요할 때 읽을 문서는 맨 아래 「문서 지도」에 있습니다.

---

## 1. 이 템플릿은 무엇인가요?

**1인 사용자가 다량의 프로젝트를 팀(사업 영역)·역할(작업 모드)로 분류해 운영하기 위한 폴더 기반 틀**입니다.
검증된 회사의 업무 프로세스를 1인 회사에 도입하는 것 — 폴더 구조와 문서 규칙만으로 "어느 영역이, 어떤 모드에서, 어디까지 됐는지"를 관리합니다. (같은 구조 그대로 여러 사람의 협업에도 쓸 수 있습니다 → 8장 끝 참조)

핵심 아이디어는 3가지입니다:

| # | 아이디어 | 쉽게 말하면 |
|---|---|---|
| 1 | **폴더가 곧 역할** | 어느 폴더에서 `claude`를 켜는지가 당신의 역할과 권한을 정합니다. 팀 폴더에서 켜면 팀 관리자, 프로젝트 폴더에서 켜면 그 프로젝트의 작업자입니다. |
| 2 | **문서가 곧 기억** | Claude의 대화 내용(컨텍스트)은 세션이 끝나면 사라집니다. 그래서 **작업을 마칠 때 문서에 저장**(`/handover`)하고, **시작할 때 문서를 읽어 복원**합니다. |
| 3 | **대시보드가 곧 현황** | 각 팀의 상태가 대시보드로 취합됩니다. 총괄자는 명령 한 번(`/report_root`)으로 전체 현황을 최신화합니다. |

## 2. 딱 3가지만 기억하세요

1. **작업은 자기 폴더에서** — 코드는 `00_Project/{프로젝트}/`, 문서는 자기 역할 폴더, 총괄은 루트.
2. **끝날 때 `/handover`** — 이걸 빼먹으면 다음 세션(내일의 나)이 아무것도 기억하지 못합니다.
3. **이름은 영문으로** — 팀명·폴더·파일명은 영문/숫자/하이픈만. 날짜는 `YYYY-MM-DD`.

## 3. 처음 5분 — 시작하기

```
1. 이 템플릿 폴더(루트)에서 터미널을 열고 claude 실행
2. /status 입력          → 현재 전체 현황 요약을 보여줍니다 (처음엔 "팀 없음")
```

루트에서 세션을 열면 당신은 **총괄자(Root 관리자)**입니다. 전체를 볼 수 있지만, 팀의 작업 파일을 직접 고치지는 않습니다.

## 4. 첫 팀 만들기 (따라하기)

루트 세션에서:

```
/new_team MyTeam
```

### 역할 구성 고르기 — 생성 전에 물어봅니다

팀을 만들기 전에 `/new_team`이 **어떤 역할로 일할지**를 확인합니다:

| 선택 | 역할 구성 | 이런 팀에 적합 |
|---|---|---|
| **표준 3역할** | `01_planner`(기획·디자인) → `02_developer`(개발) → `03_package`(배포·완료 백업) | 코드 산출물이 있는 일반적인 팀 |
| **커스텀** | 역할을 직접 정의 — 예: 창작 팀 `01_writer`(집필) → `02_editor`(교정) → `03_publisher`(발행·백업) | 글·영상·디자인 등 코드를 다루지 않는 팀 |

커스텀을 고르면 **3가지만 답하면 됩니다**:
1. **역할 목록** — 작업이 흐르는 순서대로 (영문 소문자, 폴더는 `01_`부터 번호가 붙습니다)
2. **각 역할의 임무 1줄** — "무엇을 하는 모드인가"
3. **완료 백업 담당** — 끝난 결과물을 `90_result_output/`에 보관할 역할 하나 (보통 마지막 역할)

그러면 역할마다 필요한 파일 3종(역할 정의 `CLAUDE.md` · 인수인계 `handover_{역할}.md` · 공유 접근 설정)이 **자동 생성**되고, 팀 `CLAUDE.md`의 역할 표·권한표도 실제 구성으로 맞춰집니다. 표준 일부만 바꾸는 혼합 구성(예: developer는 유지하고 `04_designer` 추가)도 가능하고, 나중에 역할을 추가·변경할 수도 있습니다 ([manual/06](manual/06_team_manager.md)의 커스텀 역할 체크리스트).

역할 구성과 함께 **「팀 목표」(1~2문장)도 물어봅니다** — 답하면 팀 `CLAUDE.md`에 바로 채워져서, 생성 직후부터 모든 세션이 팀의 방향을 읽고 시작합니다. (아직 못 정했으면 "미정"으로 두고 나중에 채워도 됩니다.)

### 생성 결과 (표준 구성 기준)

`00_Team/ProjectTeam_MyTeam/`이 아래 구조로 자동 생성됩니다 (커스텀이면 `01_~03_` 부분이 정의한 역할로 대체):

```
ProjectTeam_MyTeam/
├── CLAUDE.md              팀 관리자 정의 ← 「팀 목표」는 생성 시 입력한 내용으로 채워집니다
├── handover.md            팀 인수인계
├── 00_Project/            진행 중 프로젝트가 여기 모입니다
├── 01_planner/            기획 문서 작업 공간
├── 02_developer/          개발 기술 문서 작업 공간
├── 03_package/            빌드·배포 작업 공간
├── 10_Dashboard/          팀 현황판
├── 11_team_doc/           팀 공용 문서 (00_final 확정본 / 10_working 작업 중 / 90_old 대체본)
└── 90_result_output/      팀 최종 결과물 보관
```

생성 후 할 일: 팀 목표를 "미정"으로 뒀다면 `CLAUDE.md`의 **「팀 목표」**를 채우고 (이후 모든 세션이 이 문서를 읽습니다), 커스텀 구성이었다면 각 역할 `CLAUDE.md`의 「임무」도 한 번 다듬어 주세요.

## 5. 첫 프로젝트 만들기

팀 폴더(`00_Team/ProjectTeam_MyTeam/`)에서 새 세션을 열고:

```
/new_project my-app
```

→ `00_Project/01_my-app/`(작업 폴더)와 `91_project_process/01_my-app.md`(process 문서)가 한 쌍으로 생성됩니다.

**process 문서(`91_project_process/01_my-app.md`)는 프로젝트의 유일한 상태 문서**입니다 — 목표, 현재 단계(기획→개발→패키지), 진행 기록, 다음 할 일, 블로커가 모두 이 한 파일에 있습니다. 생성 직후 「목표 / 완료 조건」을 채우세요.
갱신은 **팀 관리자 전담**입니다 — 팀원은 자기 handover에 `[01_my-app]` 태그로 기록하면, 팀 관리자의 `/handover`가 여기에 반영합니다.
`00_Project/`의 프로젝트 폴더는 실제 앱/게임의 **독립 git 저장소**로 운영할 수 있습니다 (템플릿 저장소는 `00_Project`를 추적하지 않고, 프로젝트 상태는 process 문서가 담습니다).

## 6. 일하는 방법 — 모든 세션의 시작과 끝

어떤 작업이든 패턴은 같습니다:

```
[시작]  그 폴더의 CLAUDE.md + 상태 문서(handover 또는 process 문서)를 읽는다
        → Claude가 맥락 요약을 보여주면, 「다음 할 일」 1번부터 시작
[작업]  파일은 모두 그 폴더 안에서만
[종료]  /handover 실행 → 상태 문서 갱신 → 커밋
```

무슨 작업을 어디서 하는지:

| 하고 싶은 것 | claude를 켜는 곳 | 상태 문서 |
|---|---|---|
| 기획·디자인 작업 | `01_planner/` | `handover_planner.md` |
| 개발(코드) 작업 | `02_developer/` — 코드는 공유 `../00_Project/{프로젝트}/`에 | `handover_developer.md` |
| 배포·최종 선정 작업 | `03_package/` — 완료 백업은 `../90_result_output/`에 | `handover_package.md` |
| 팀 조율·process 갱신·구현 지휘 | 팀 폴더 | `handover.md` + 각 process 문서(`91_project_process/`) |
| 전체 현황·팀 생성 | 루트 | `handover_root.md` |

커스텀 역할 팀도 패턴은 같습니다 — **자기 역할 폴더에서 켜고, 상태 문서는 `handover_{역할}.md`** (예: `01_writer/` → `handover_writer.md`). 위 표의 기획/개발/패키지 행을 자기 역할로 바꿔 읽으면 됩니다.

**팀 내 상호 호환**: 팀원은 자기 역할 폴더에서 세션을 열지만, 공유 4폴더(`00_Project`·`11_team_doc`·`90_result_output` + `91_project_process` 읽기)에는 접근할 수 있습니다 (역할 폴더의 설정이 자동 허용). 원칙은 하나입니다 — **메인 작업은 프로젝트 폴더(`00_Project/{프로젝트}/`)에서, 보조 작업(초안·메모·역할 문서)은 내 역할 폴더에, 기록은 내 handover에.** 문서 위치가 헷갈리면 **프로젝트 산출물=`00_Project` → 팀 공유 확정본=`11_team_doc/00_final` → 보조=내 폴더** 순으로 판단하세요.

> ⚠ **코드 수정은 `00_Project/{프로젝트}/` 안에서만** — 주체는 개발 역할과 팀 관리자입니다.
> ⚠ **process 문서는 직접 고치지 않습니다** — handover에 `[NN_프로젝트명]` 태그로 기록하면 팀 관리자가 반영합니다.
> ⚠ 타 역할 폴더·상위 폴더·다른 팀에는 접근하지 않습니다 (격리 원칙).

작업 흐름은 **기획 → 개발 → 패키지** 순서로 흐르고, 다음 사람에게 넘길 일은 handover의 「다음 할 일」에 태그와 함께 남기면 됩니다.

## 7. 운영 루틴 (역할별)

**팀원 (매 세션)**
```
자기 폴더에서 세션 → 작업 → /handover → 커밋
```

**팀 관리자 (수시 + 주 1회)**
```
1. 팀 폴더에서 91_project_process/의 process 문서와 역할 handover들을 읽고 조율
2. /report        → 팀 보고서 + 팀 대시보드 갱신
3. /handover      → 팀 인수인계 취합
```

**총괄자 (주 1회 이상, 루트에서)**
```
1. /report_root   → 전 팀 보고 일괄 수거 + 전체 대시보드 갱신
                    (HTML 대시보드도 자동 재생성됨)
2. /status        → 요약 확인 (지연된 팀엔 ⚠STALE 표시)
3. /handover_root → 전체 인수인계 취합 (필요 시)
```

**웹 대시보드로 보기**: `10_Dashboard/dashboard.html`을 브라우저로 열면 전체 현황을 한눈에 볼 수 있습니다 (좌측에서 팀 클릭 → 팀 상세). 각 팀 폴더의 `10_Dashboard/dashboard.html`은 그 팀만 보여줍니다. 수시 재생성은 루트에서 `/dashboard` — **HTML은 열람 전용**이고 내용 수정은 항상 handover/process에 합니다.

## 8. 기본 사용 모델 — 1인 회사 모드

> 이 템플릿의 기본 사용자는 **1인**입니다 — 검증된 회사의 업무 프로세스를 1인 회사에 도입합니다.
> 팀원을 뽑는 게 아니라 — **팀은 사업 영역으로 나누고, 역할은 시간대로 나눠 맡습니다.**
> 격리·전담 규칙은 1인일 때 관료제가 아니라 **세션 컨텍스트 오염 방지 + 사고 모드 분리 장치**입니다.

### 2축 모델

| 축 | 의미 | 예 |
|---|---|---|
| **팀 = 무엇을** (도메인) | 오래 지속되는 사업 영역마다 팀 하나 | `ProjectTeam_WebDev`(웹개발) · `ProjectTeam_Creative`(창작물) · `ProjectTeam_Contents`(컨텐츠) · `ProjectTeam_DataOps`(데이터 관리) |
| **역할 = 어떤 모드로** (모자 바꿔쓰기) | 역할 폴더에서 세션을 열면 그 역할의 일만 한다 | `01_planner`에서 기획 → `02_developer`에서 구현 → `03_package`에서 마무리 |

그리고 **루트 = 나의 주간 포트폴리오 리뷰**가 됩니다. `/report_root` 한 번이면 대시보드에 내 모든 영역의 현황이 모이고, 방치한 영역에는 ⚠STALE이 떠서 "조용히 썩는 사이드 프로젝트"를 잡아줍니다.

### 사이클

```
시작    팀 폴더(관리자 모드) — process 문서 읽고 이번에 맡을 역할 결정
작업    해당 역할 폴더에서 세션 — 그 역할의 일만 한다
종료    /handover (태그 필수) → 다음 역할 또는 내일의 나에게 전달
정기    팀 폴더 /handover 취합 → 루트 /report_root (포트폴리오 리뷰)
```

역할별 handover는 **나에게 보내는 업무 지시서**입니다. "기획자인 내"가 `[01_my-app]` 태그로 남긴 다음 할 일을 "개발자인 내"가 받습니다.

### 시간 패턴별 변형 — 자기 생활에 맞는 것을 고르세요

| 패턴 | 이런 분 | 운영 방식 |
|---|---|---|
| **하루형** (풀타임) | 종일 작업 가능 | 위 사이클 그대로 — 역할 전환은 하루 1~2회, 반나절 블록 |
| **파편형** (1~2시간 조각) | 학생·본업 병행 | **1조각 = 1역할** 또는 **하루 1팀** — 조각 안에서 역할·팀을 바꾸지 않습니다 |
| **주간형** (저녁·주말만) | 부업·사이드 | **주 단위로 역할 고정** ("이번 주는 개발 주간") — 포트폴리오 리뷰는 격주 |

포트폴리오 리뷰 주기는 투입 시간에 비례해 조정하세요 (주 1회 ~ 격주). 어느 패턴이든 원칙은 하나입니다: **한 세션에서는 한 가지 모드만.**

### 1인일 때 줄여도 되는 것 / 줄이면 안 되는 것

| 줄여도 됨 | 줄이면 안 됨 |
|---|---|
| 루트 총괄자 루틴 — 주 1회 리뷰면 충분 | `/handover` — 유일한 기억 장치 |
| 휴면 팀의 `/report` | 태그 `[NN_프로젝트명]` — 관리자 모드의 내가 취합할 근거 |
| `11_team_doc` 회의록류 | process 문서 관리자 전담 — 작업 모드에서 직접 수정 금지는 1인일 때 더 중요 |

### 가드레일 5가지

1. **팀 기준**: 팀 = 오래 지속되고 컨벤션이 다른 사업 영역, 프로젝트 = 완료 조건이 있는 산출물. 프로젝트가 하나뿐이고 앞으로도 그럴 것 같으면 팀을 만들지 말고 기존 팀의 프로젝트로 두세요.
2. **미리 다 만들지 말 것**: 활동 중인 1~2팀으로 시작하고 필요할 때 `/new_team`. 빈 팀은 대시보드 노이즈이자 관리 세금입니다.
3. **휴면 선언**: 의도적으로 쉬는 팀은 팀 handover의 「현재 상태」에 **휴면**이라고 적어두세요 (STALE 오탐 방지).
4. **팀 간 공유는 루트 결과물로**: 다른 팀이 상시 참조할 것은 `11_doc_result`로 승격합니다. 타 팀 폴더 직접 접근 금지는 혼자여도 유지 — 이 격리가 깨지면 템플릿의 전제가 무너집니다.
5. **한 세션 한 모드**: 잦은 역할 전환은 `/handover`를 세금으로 만듭니다. 시간 블록의 크기는 위 「시간 패턴별 변형」 표대로 자기 생활에 맞추되, 한 세션 안에서 역할·팀을 바꾸지 않는 원칙만 지키세요.

> 표준 3역할이 안 맞는 팀(예: 창작물)은 **팀을 만들 때 `/new_team`의 역할 구성 질문에서 커스텀**을 고르세요 — 역할 파일이 자동 생성됩니다 (4장). 이미 만든 팀의 역할 추가·변경은 [manual/06_team_manager.md](manual/06_team_manager.md)의 3종 체크리스트를 따릅니다.

### 다인 협업으로 확장

역할 폴더를 실제 사람에게 배정하면 그대로 다인 팀이 됩니다 — 규칙·구조·명령은 동일하고, "모자 바꿔쓰기"가 "사람 간 인수인계"로 바뀔 뿐입니다.

## 9. 자주 쓰는 명령 한눈에

| 명령 | 어디서 | 무엇을 |
|---|---|---|
| `/new_team {팀명}` | 루트 | 팀 생성 — **역할 구성(표준/커스텀) 확인 후** 자동 스캐폴딩 |
| `/new_project {이름}` | 팀 폴더 | 프로젝트 생성 (`00_Project/NN_이름/` 작업 폴더 + `91_project_process/NN_이름.md` process 문서) |
| `/handover` | 작업한 폴더 | 상태 문서 갱신 (세션 마무리 필수!) |
| `/report` | 팀 폴더 | 팀 보고서 + 팀 대시보드 갱신 |
| `/report_root` | 루트 | 전 팀 보고 + 전체 대시보드 갱신 |
| `/handover_root` | 루트 | 전 팀 인수인계 취합 (`handover_root.md`) |
| `/status` | 루트/팀 폴더 | 현황 요약 조회 (수정 없음) |
| `/dashboard` | 루트 | HTML 대시보드 재생성 (관리자 + 전 팀 페이지) |
| `/pipeline {주제}` | 어디서나 | 탐색→설명→검증→권고 4단계 심층 분석 |

## 10. 자주 하는 실수 (이것만 피하세요)

- ❌ `/handover` 없이 세션 종료 → 다음 세션이 맥락을 잃습니다. **문서가 곧 기억**입니다.
- ❌ 팀원이 process 문서를 직접 수정 → 팀 관리자 전담입니다. handover에 `[NN_프로젝트명]` 태그로 기록하세요.
- ❌ 태그 없는 프로젝트 기록 → 프로젝트 현황에 반영되지 않습니다.
- ❌ 대시보드를 손으로 수정 → 다음 `/report`가 덮어씁니다.
- ❌ 개발 역할이 아닌데 코드 수정 → 코드는 `00_Project/{프로젝트}/`에서 개발·팀 관리자만.
- ❌ 팀 폴더를 손으로 생성 → 표준이 깨집니다. 반드시 `/new_team`으로.
- ❌ 프로젝트 폴더에 CLAUDE.md·handover.md·process.md 추가 → 프로젝트 상태 문서는 `91_project_process/`의 process 문서 하나입니다.
- ❌ 한글 폴더·파일명 → 영문만 (한글은 특이한 경우에 한해 최대한 짧게).
