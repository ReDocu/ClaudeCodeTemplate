# 메타프롬프트 — 기능 상세 기술문서 생성기

> **용도**: 임의의 코드베이스에 대해 `Tech_document.html` 수준의 **상세 기술문서**를 생성시키는 프롬프트.
> 그래픽/비주얼 스타일 지시가 아니라 **내용의 상세도와 정확성**을 강제하는 것이 목적이다.
>
> **사용법**: 아래 구분선 사이의 프롬프트를 복사 → `{{ }}` 자리를 채움 → 대상 저장소를 연 Claude 세션에 붙여넣기.

---

## ⬇ 프롬프트 시작

당신은 이 코드베이스를 처음 여는 개발자가 **문서만 읽고 구조·동작·경계를 정확히 파악**할 수 있게 만드는 시니어 아키텍트 겸 테크니컬 라이터다. 아래 규격에 따라 기술문서를 작성하라.

### 0. 입력

| 항목 | 값 |
|---|---|
| 대상 코드베이스 | `{{저장소 루트 경로 — 현재 열린 저장소면 "이 저장소"}}` |
| 버전/릴리스 | `{{버전 — 모르면 git tag·changelog에서 실측}}` |
| 출력 파일 | `{{파일명.html 또는 .md}}` |
| 출력 형식 | `{{단일 HTML(인라인 SVG 다이어그램) | Markdown(mermaid)}}` |
| 함께 볼 문서 | `{{README·기능명세·매뉴얼 등 — 없으면 생략}}` |
| 독자 | 이 코드를 처음 여는 개발자 (마케팅·경영진 아님) |

### 1. 절대 원칙 — 실측주의 (이것이 문서의 정체성이다)

1. **모든 문장은 코드에서 직접 확인한 사실만 쓴다.** 문서를 쓰기 전에 반드시 실제 파일을 열어 읽는다. 기억·추론·일반론으로 채우지 않는다.
2. **추측 표현 전면 금지** — "아마", "~일 것이다", "일반적으로", "보통" 이 나오면 그 문장은 삭제 대상이다. 확인 못 한 것은 쓰지 말고, 확인했는데 불확실한 것은 "알려진 경계" 섹션으로 보낸다.
3. **구체 수치 의무화** — TTL, 폴링 주기, 재시도 횟수×간격, 타임아웃, 포트, 상한(body 크기 등), 파일별 라인 수를 **실측값 그대로** 쓴다. "주기적으로 갱신"(✕) → "TTL 1.5s 캐시, stale 허용"(○). "여러 번 재시도"(✕) → "≤6회 × 500ms"(○).
4. **실제 식별자만 사용** — 함수명·모듈 경로·엔드포인트·설정 키·label 형식을 코드에 있는 철자 그대로 쓴다. 코드에 없는 이름을 지어내면 그 문서는 실패작이다.
5. **날짜 스탬프** — 제목 부근과 말미에 "YYYY-MM-DD 코드 실측 기준"을 명시한다. 문서는 스냅샷이지 영원한 진실이 아니다.
6. **설계 이유를 한 줄씩 싣는다** — 규칙·구조를 서술할 때 "어기면 무슨 버그가 재발하는가"를 붙인다. (예: "stale 상태로 스폰을 결정하면 중복 pane이 생긴다")

### 2. 상세도 규격 — 기능 하나를 서술할 때 답해야 하는 6문항

기능·흐름 하나를 서술하는 단락은 아래 여섯을 모두 답할 때까지 완성이 아니다:

| 질문 | 예시 답 형태 |
|---|---|
| **무엇을** 하는가 | 동작 1문장 |
| **어디서** 하는가 | `모듈경로 → 함수명()` — 진입점은 **굵게** |
| **언제** 실행되는가 | 트리거(요청/훅/루프) · 주기 · 조건 가드 `[...]` |
| **어떤 값**으로 | 수치·형식·payload 필드 |
| **실패하면** | 오류 코드(409/400 등)·폴백·자기치유 경로 |
| **왜** 이렇게 했는가 | 설계 이유 또는 어겼을 때의 버그 1줄 |

### 3. 문서 골격 — 12섹션 (프로젝트 성격에 맞게 가감하되, 빼면 사유를 문서 안에 남긴다)

1. **프로그램 개요 · 현황** — 핵심 컨셉 1문단 + 항목 표(계층 매핑·진실의 원천·전제 환경·보안 경계) + 최신 버전 변경점 목록
2. **시스템 아키텍처** — 컴포넌트 다이어그램 1개: 계층(진입/프레젠테이션/도메인/프로브/외부)과 호출 방향, 화살표마다 실제 함수명 라벨
3. **폴더 구조** — 트리 코드블록, **모든 항목에 한 줄 주석**(역할이 뭔지), 정본/폐기/gitignore 구분 표시
4. **모듈 구조** — 모듈(또는 클래스)별 카드: «stereotype» / 책임 1줄 / 공개 연산 목록(최다 호출 진입점 굵게) + **의존 방향 표**(누가 무엇을 import 하는가, 비고에 규칙)
5. **동작 시퀀스** — 핵심 흐름 3~4개(부팅·주 폴링/조회·주 변이·이벤트 파이프라인)를 시퀀스 다이어그램으로. 각 화살표에 함수명·조건 `[가드]`·실패 분기 표기
6. **상태 전이** — 상태머신: 상태 목록·전이 트리거(엔드포인트/이벤트)·금지 전이(오류 코드까지)
7. **데이터 모델 · 런타임 파일** — 핵심 선언 파일과 API payload를 **실제 JSON 예시 + 필드별 주석**으로. 런타임 생성 파일은 "쓰는 곳 → 읽는 곳" 표로
8. **API/인터페이스 요약** — 분류별 표(조회/변이/운영), 인증·상한·confirm 게이트 같은 공통 규약을 표 밖 1문장으로
9. **캐시 · 신뢰성 설계** — 캐시 지도 표(대상·TTL·모듈·비고) + 신뢰성 기전 목록(자기치유·재검증·성능 저하 시 동작)
10. **플랫폼/환경 추상화** — 추상화 계약과 구현체 대비 표 (해당 없으면 "설정·환경 분기"로 대체)
11. **보안 · 불변 규칙** — 보안 경계 목록 + **불변 규칙 번호 목록**("어기면 재발하는 버그" — 코드 실측으로 확정된 것만)
12. **검증 컨벤션 · 코드 규모 · 알려진 경계** — 테스트/검증 방법 실태, 파일별 라인 수 실측 표, 미구현·비목표·알려진 한계 (숨기지 말 것 — 경계를 쓰는 문서가 신뢰를 얻는다)

### 4. 표현 규칙 — 형식은 정보 밀도를 위해서만 쓴다

- **표**를 쓰는 곳: 의존 방향 · 캐시 지도 · API 요약 · 플랫폼 대비 · 런타임 파일. 셀은 짧게, 설명은 비고 열이나 주변 문장에.
- **코드블록**을 쓰는 곳: 폴더 트리 · payload 예시. payload는 반드시 필드마다 `// 주석`을 단다.
- **다이어그램은 장식이 아니라 정보다** — 모든 박스에 실제 모듈명, 모든 화살표에 실제 함수명 또는 조건 라벨. 라벨 없는 화살표는 그리지 않는다. 외부 컴포넌트는 점선 등으로 구분. (비주얼 테마·색은 별도 스타일 템플릿의 몫 — 이 프롬프트는 관여하지 않는다)
- **callout(note)**: 문서 전체에서 가장 중요한 데이터 계약 1~2개만 격상한다. (예: "핵심 계약은 `GET /api/state` 페이로드 하나다")
- **강조 규칙**: 진입점 함수·핵심 원칙만 굵게. 전부 굵으면 아무것도 굵지 않은 것이다.
- **상호 참조**: 형제 문서(README·기능명세·매뉴얼)가 있으면 서두와 말미에 역할 분담을 명시한다 ("기능 단위 명세는 X를, 사용법은 Y를 본다").

### 5. 작성 절차 — 4패스

1. **Pass 1 실측 스캔**: 진입점부터 따라가며 전 모듈을 읽는다. 수치(TTL·주기·상한·라인 수)·식별자·엔드포인트 목록을 수집한다. 라인 수는 실제로 센다.
2. **Pass 2 골격 확정**: §3의 12섹션을 이 프로젝트에 맞게 가감하고 목차를 만든다.
3. **Pass 3 본문 작성**: 섹션마다 §2의 6문항 규격으로 채운다. 다이어그램은 텍스트로 흐름을 먼저 확정한 뒤 그린다.
4. **Pass 4 검증**: 아래 체크리스트로 자체 검수하고, 실패 항목은 고친 뒤 제출한다.

### 6. 셀프 검증 체크리스트 (하나라도 걸리면 미완성)

- [ ] 문서에 등장하는 모든 함수명·파일 경로·엔드포인트가 코드에 실재하는가? (grep으로 대조)
- [ ] 수치 없이 "주기적으로/여러 번/큰/작은" 같은 모호어로 때운 단락이 없는가?
- [ ] 추측 표현("아마", "~일 것") 이 0건인가?
- [ ] 각 다이어그램의 화살표에 전부 라벨이 있는가?
- [ ] 불변 규칙 각각에 "어기면 생기는 버그"가 붙어 있는가?
- [ ] 미구현·비목표·알려진 한계 섹션이 비어 있지 않은가? (완벽해 보이는 문서는 의심된다)
- [ ] 날짜 스탬프와 형제 문서 상호 참조가 서두·말미에 있는가?

### 7. 금지사항

- 마케팅 문구, 기능 자랑, "강력한/유연한/간편한" 류의 형용사
- 코드에 없는 이름의 창작, 확인 안 한 동작의 서술
- "등", "기타" 로 목록을 얼버무리기 — 유한 목록은 끝까지 나열한다
- 일반 소프트웨어 상식 설명 (HTTP가 뭔지, 캐시가 뭔지 — 독자는 개발자다)
- 내용 없이 형식만 채우기 — 섹션 제목만 있고 실측이 없는 섹션은 만들지 않는다

### 참고 — 이 규격의 원본

이 메타프롬프트는 `Tech_document.html`(ClaudeCockpit 기술문서, 2026-07-20 코드 실측 기준)의 작성 규율을 역추출한 것이다. 원본에서 상세도의 핵심은 그래픽이 아니라:

- 모든 캐시에 TTL 실측값 (1.5s / 4s / 15s / 30s / 2.5s)
- 모든 흐름에 함수명 + 조건 가드 (`③ verifyPlacement — refreshState ≤6회 × 500ms`)
- 불변 규칙마다 위반 시 버그 명시 (`stale 스폰 결정 → 중복 pane`)
- 파일별 라인 수 실측 표 + 알려진 경계의 정직한 나열

이었다. 스타일(다크 테마·SVG 팔레트)이 필요하면 별도 그래픽 템플릿 프롬프트와 조합해서 쓴다.
