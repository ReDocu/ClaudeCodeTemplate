# 매뉴얼 — 개별 관리자 (역할 담당자)

> 대상: 팀 안에서 하나의 역할(기획·디자인 / 개발 / 패키지 또는 커스텀 역할)을 맡아 실제 작업을 수행하는 사람.

## 1. 당신은 누구인가

- **세션 위치**: 항상 **자기 역할 폴더**에서 `claude`를 실행한다.
  - 기획·디자인: `00_Team/{팀폴더}/01_planner/`
  - 개발: `00_Team/{팀폴더}/02_developer/`
  - 배포·최종 선정: `00_Team/{팀폴더}/03_package/`
  - 커스텀 역할: 해당 `NN_{역할}/` (CLAUDE.md + `handover_{역할}.md`가 있어야 한다 — 없으면 팀 관리자에게 요청)
- **공유 폴더 접근 (상호 호환)**: 역할 폴더의 `.claude/settings.json`이 팀 공유 폴더 접근을 허용한다. 접근 확인 프롬프트가 뜨면 아래 범위 안에서 승인해도 된다.

| 공유 폴더 | 권한 |
|---|---|
| `../00_Project/{프로젝트}/` | ✅ 작업물 쓰기 (**코드 수정은 개발 역할만**) |
| `../00_Project/*/process.md` | 📖 읽기만 — 갱신은 팀 관리자 전담 |
| `../11_team_doc/` | ✅ 쓰기 — 공유 문서 취합 |
| `../90_result_output/` | 패키지만 쓰기, 나머지는 읽기 |
| 타 역할 폴더 · `10_Dashboard` · 상위/타 팀 | ❌ 접근 금지 |

## 2. 작업 사이클

```
1. 세션 시작   → 내 CLAUDE.md(역할 정의)와 handover_{역할}.md 를 읽는다
               (프로젝트 상황이 필요하면 ../00_Project/{프로젝트}/process.md 도 읽는다)
2. 작업        → 메인 작업은 ../00_Project/{프로젝트}/ 에서 (산출물의 본체),
               보조 작업(초안·메모·역할 문서)은 내 폴더에, 공유 확정본은 ../11_team_doc/00_final/ 에
3. 세션 종료   → /handover 실행 → handover_{역할}.md 갱신
               ★ 프로젝트 관련 항목엔 반드시 [NN_프로젝트명] 태그
4. 커밋        → docs(handover): {팀명}/{역할}
```

**문서 위치 결정 흐름** — 저장할 때마다 이 순서로 판단한다:
① 특정 프로젝트의 산출물(**메인 작업물**) → **`../00_Project/{프로젝트}/`** — 작업의 본체는 여기서 진행한다
② 팀이 참조할 공유 문서 → **`../11_team_doc/`** — 확정본은 `00_final/`, 검토 중이면 `10_working/`, 대체된 구버전은 `90_old/`로 (기준은 중요도가 아니라 **수명 단계**)
③ 초안·메모 등 내 역할만 보는 **보조 작업물** → **내 폴더**
헷갈리면 두 번 물어라: "특정 프로젝트에 속하나?" → ①, "다른 역할이 찾을 일이 있나?" → ②, 둘 다 아니면 ③.

**인수인계 받기**: `handover_{역할}.md`를 목표 → 현재 상태 → 다음 할 일 순으로 읽고 1번부터 시작한다. 5분이면 충분해야 한다 — 부족하면 이전 담당자에게 보완을 요청.

## 3. 태그 규칙 — 내 기록이 process.md가 되는 경로

process.md는 직접 고치지 않는다. 대신 handover 항목에 태그를 달면 **팀 관리자가 `/handover` 취합 때 process.md에 반영**한다:

```markdown
## 완료된 것
- [01_my-app] 결제 플로우 기획서 v2 완료 (00_Project/01_my-app/design/flow-v2.md)

## 진행 중인 것
- [01_my-app] 재시도 로직 구현 — 테스트 12/16 통과 (07-08 기준)

## 다음 할 일
1. [01_my-app] 백오프 상한 처리   ← 다음 역할·다음 세션에 넘길 일
```

태그가 없는 항목은 프로젝트 현황에 반영되지 않는다 (역할 일반 업무로 취급).

## 4. 역할 간 전달 (작업 흐름: 기획 → 개발 → 패키지)

- 다음 역할로 넘길 것은 「다음 할 일」에 태그와 함께 우선순위 순으로.
- 개발 → 패키지: 결과물의 **경로와 상태** 명시.
- 패키지 → 개발(결함 반송): 결함을 태그와 함께 「다음 할 일」에 — **소스 직접 수정 금지**.
- 패키지의 완료 백업: 최종 선정한 작업물을 `../90_result_output/`에 보관 (패키지 역할만).

## 5. 명령 레퍼런스

| 명령 | 실행 위치 | 하는 일 |
|---|---|---|
| `/handover` | 내 역할 폴더 | `handover_{역할}.md` 갱신 (태그 기록) |
| `/pipeline {주제}` | 내 역할 폴더 | 4단계 분석 (결과물은 현재 폴더 또는 `../11_team_doc/`) |

※ `/new_project`는 팀 폴더 전용 — 필요하면 팀 관리자에게 요청. process.md 갱신도 팀 관리자의 몫.

## 6. 하면 안 되는 것

- `process.md` 직접 수정 — 팀 관리자 전담. 내 기록은 handover 태그로.
- 개발 역할이 아닌데 코드 수정 (기획=문서·디자인, 패키지=검증·빌드·백업).
- 타 역할 폴더·`10_Dashboard`·상위/타 팀 접근.
- 태그 없이 프로젝트 작업 기록 — 프로젝트 현황에서 누락된다.
- handover 갱신 없이 세션 종료 — 문서가 곧 기억이다.
- 추측으로 문서 채우기 — 확인 안 된 항목은 "(미확인)".
- 한글 파일명 (특이 경우에 한해 최대한 짧게).
