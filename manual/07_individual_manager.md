# 매뉴얼 — 개별 관리자 (역할 담당자)

> 대상: 팀 안에서 하나의 역할(기획/개발/패키지 또는 커스텀 역할)을 맡아 실제 작업을 수행하는 사람.

## 1. 당신은 누구인가

- **세션 위치**: 자기 역할 폴더에서 `claude`를 실행한다.
  - 기획: `00_Team/{팀폴더}/01_planner/`
  - 개발: `00_Team/{팀폴더}/02_developer/`
  - 패키지: `00_Team/{팀폴더}/03_package/`
  - 커스텀 역할: 해당 `NN_{역할}/` 폴더 (CLAUDE.md + `handover_{역할}.md`가 있어야 한다 — 없으면 팀 관리자에게 요청)
- **권한**: 자기 역할 폴더 안에서만 작업한다. **코드 수정은 `02_developer`에서만** — 기획·패키지는 문서/빌드 산출물 전용.
- **격리**: 상위 폴더(팀 루트·타 역할·루트)에 접근하지 않는다. 타 역할의 정보가 필요하면 사용자에게 해당 handover 공유를 요청한다.

## 2. 작업 사이클

```
1. 세션 시작   → 폴더의 CLAUDE.md(내 역할 정의)와 handover_{역할}.md 를 읽는다
2. 작업        → 「다음 할 일」 1번부터. 작업 파일은 모두 이 폴더 안에
3. 세션 종료   → /handover 실행 → handover_{역할}.md 갱신 (덮어쓰기)
4. 커밋        → docs(handover): {팀명}/{역할}
```

**인수인계 받기**: 새로 역할을 맡았다면 `handover_{역할}.md`를 목표 → 현재 상태 → 다음 할 일 순으로 읽고, 「다음 할 일」 1번부터 시작한다. 5분이면 충분해야 한다 — 부족하면 이전 담당자에게 handover 보완을 요청.

## 3. 역할 간 전달 (작업 흐름: 기획 → 개발 → 패키지)

- 다음 역할로 넘길 것은 내 `handover_{역할}.md`의 **「다음 할 일」에 우선순위 순으로** 남긴다.
- 개발 → 패키지: 넘길 결과물의 **경로와 상태**를 명시한다.
- 패키지 → 개발(결함 반송): 발견한 결함을 「다음 할 일」에 남긴다 — **직접 수정 금지**.
- 팀 관리자가 팀 폴더에서 `/handover`를 실행하면 각 역할 handover가 팀 `handover.md`로 취합된다.

## 4. 명령 레퍼런스

| 명령 | 실행 위치 | 하는 일 |
|---|---|---|
| `/handover` | 역할 폴더 | `handover_{역할}.md` 갱신 (하위 프로젝트 요약 포함) |
| `/new_project {이름}` | 역할 폴더 | 작업이 커질 때 하위 프로젝트 폴더 생성 (주로 02_developer) |
| `/handover` | 프로젝트 폴더 | 프로젝트 `handover.md` 갱신 |
| `/pipeline {주제}` | 역할/프로젝트 폴더 | 4단계 분석 (결과물은 현재 폴더에 저장) |

## 5. 하면 안 되는 것

- 상위 폴더(팀 루트·타 역할) 파일 접근·수정 — 팀 문서 반영은 팀 관리자의 `/handover` 취합에 맡긴다.
- `02_developer` 밖에서의 코드 수정.
- handover 갱신 없이 세션 종료 — 문서가 곧 상태다. 다음 사람(혹은 내일의 나)은 handover만 읽는다.
- 추측으로 handover 채우기 — 확인 안 된 항목은 "(미확인)"으로 남긴다.
- 한글 파일명 (특이 경우에 한해 최대한 짧게).
