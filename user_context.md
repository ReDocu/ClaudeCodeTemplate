## 1. 최초 요구사항 초안 (2026-07-12, 재기획 킥오프 — 원문)

### 요구사항

1. wmux의 세션을 활용하여 개발을 하는데 있어, 프로젝트 관리가 어렵다.
2. 대시보드를 통해 기존 프로젝트를 wmux에 연동, 관리하는 프로그램을 구현하고자 한다.
3. 프리랜서 개발자 또는 1인 개발자가 자신이 개발하고 있는 프로젝트를 대시보드에서 관리하고자 한다.
4. 대시보드의 ops 폴더에서 깃, DB, 배포를 확인할 수 있는 release 프로젝트가 들어가 있어야 한다.
5. 관리자[wmux 사용자] − 프로젝트[워크스페이스] − 역할[세션] 으로 나누어 프로젝트(워크스페이스)를 관리한다.

### 사용자 맥락 (사용 흐름)

1. exe 파일 실행
2. 서버 실행 (서버 콘솔에서 오류 상태 확인)
3. 대시보드 실행 (크롬에서 대시보드 상태 확인)
4. 활성화 중인 wmux를 중심으로 대시보드 갱신
5. 대시보드에 **진행중 / 대기중 / 종료된** 프로젝트를 나열, **활성화시킨 프로젝트만 wmux에 워크스페이스 생성**
6. 프로젝트 관리 내역을 로그로 남김
7. 대시보드에서 ops는 **고정 세션**으로 wmux의 **첫번째 세션(터미널)**과 연결 관리 가능
8. 폴더 원격 접속 — 대시보드에서 생성 시, 폴더 경로는 **프로젝트/역할**로 지정
9. root 안에 만들어진 프로젝트와 템플릿 간의 운영 방식·체계·설정은 **격리**되어야 한다

→ **PRD 반영**: Part I 전체 (§1 요약 · §5 페르소나 · §6 흐름 A~E · R2/R3/R8 · F10 격리)

## 2. 오픈 퀘스천 답변 (구 handover §4의 7건 — R1~R8의 원천)

### 2-1. 직접 답변 (원문 요지)

- **① 기존 프로젝트 연동**: "root로 이사시키기"
- **② ops의 범위**: "프로젝트별 workspace 1번 세션"
- **③ 생명주기**: "비활성화시 세션 kill / 활성화시 — 터미널 실행 [클로드코드가 실행되어 있지 않을시 클로드코드 실행 버튼 / 종료 상태 확인해서 버튼 생성 유무]"
- **④ DB·배포**: "연동된 서비스 링크 / 로컬 폴더 실행하기 / 예외처리 또는 에러·버그 발생 시 서버에 로그 올려주기"

### 2-2. 후속 선택지 답변 (AskUserQuestion, 원문 선택)

- **구현 방식**: **리라이트** (기존 teamctl/대시보드는 참고용 — 증분 개정 아님)
- **"종료된 프로젝트"의 의미**: **아카이브** (폴더·선언 유지, 종료 섹션 표시, 재활성화 가능)
- **이사 수행 주체**: **대시보드 자동 이동** (경로 지정 → 폴더 move + 선언 스캐폴드 자동)
- **로그 범위**: **관리 이벤트 + 에러 통합** (중앙 JSONL, 대시보드 노출)

→ **PRD 반영**: §3 결정 레지스터 **R1~R8** (③의 "클로드코드 실행 버튼"으로 구 D18(claude 릴리스 제외) 폐기 = R4)

## 4. 설계 문답 (사용자 질문 → 확인된 방향)

### 4-1. "카드는 현재 배포중인 애들은 자동으로 만들어지는 거 맞지?"

**확인된 동작** — 기준은 "실행 중"이 아니라 **"root/에 선언됨"**:
- 자동 O: `root/<프로젝트>/project.json` 있으면 카드 자동 생성(생성·이사 즉시). 활성 프로젝트의 세션·claude on/off는 wmux 실측 폴링으로 자동 갱신, 세션 누락 경고 자동, dev 서버 포트는 포트맵에 자동 귀속.
- 자동 X: **선언 없이 wmux에만 존재하는 workspace/세션은 카드 미생성** — "선언이 진실" 구조. 선언 밖 자동 등재(구 drift/adopt)는 §13에서 의도적 제외(구 B1·B4 버그의 진원지).

### 4-2. "세션 추가 기능은 넣는 게 좋아? 아니면 wmux에 맡기는 게 좋아?"

**권고안 제시됨 (하이브리드)** → ✅ **확정** (사용자 지시 "user_context 기반 개발 내용 갱신"으로 채택 — FS-8에 [＋ 역할]/역할 제거 반영):
- **자유 스폰 버튼은 미도입 유지** — 세션 생성 경로 1개(활성화 수렴)·kill 경로 1개(비활성화 일괄)라는 리라이트 단순화가 무너짐(중복 판정 복잡화·ops 보호 와해).
- **[＋ 역할] 버튼은 도입 가치 있음** — 선언 경유 추가(FS-8 역할 병합 → FS-5 세션 복구)라 구조 무해. 백엔드 명세는 이미 존재, 카드 UI 버튼만 추가하면 됨.
- 원칙: **영속적 팀 구성 변경(역할 추가) = 대시보드 · 즉흥적 임시 터미널 = wmux.**

### 4-3. "cockpit은 root 안의 프로젝트와 독립적 운행(격리 상태)인 것 맞지?"

**확인됨 — 양방향 격리가 설계 전제** (요구9 → FS-8, 구 D16 계승):
- **cockpit → 프로젝트 (불간섭)**: 프로젝트 폴더 개입은 생성/이사 시점 스캐폴드 1회뿐(멱등·기존 파일 미덮어쓰기), 이후 프로젝트 CLAUDE.md 절대 불수정(FS-8-3). 운영 중 개입은 전부 wmux 세션 수준(스폰·claude 명령 전송·확인 경유 kill) — 프로젝트 파일·코드·설정 불관여. 관측은 읽기 전용(caps는 이름·종류만, 포트는 귀속 표시만).
- **프로젝트 → cockpit (정책 오염 차단)**: 프로젝트 자체 CLAUDE.md가 상위 cockpit 정책을 명시 무효화(조상 로드는 차단 불가 — 이 선언이 유일한 차단선) + 자체 git init으로 경계 고정.
- **격리처럼 안 보이는 예외의 해명**: `project.json`은 프로젝트 폴더 내 사이드카이나 프로젝트 git에서 비추적(머신 로컬 상태) · cockpit 런타임(config·로그)은 전부 `cockpit/workspace/` — 프로젝트 폴더에 로그를 남기지 않음.

| C4 | claude 작업 중인 프로젝트를 실수로 비활성화 | ◐ | FS-6 다이얼로그가 claude on 세션 경고 — 방어선은 있으나 **kill 후 undo 불가**(정직 표기 필요) |
| C5 | 여러 프로젝트 동시 활성화 — 세션 수·usage 급증 | ✅ | 게이지(FS-16)·ministat이 가시화 — 제한은 비목표(사용자 판단) |
| C6 | ops 세션에서 dev 서버 실행 → 포트맵 귀속 확인 | ✅ | FS-14 귀속(프로세스 cwd 기준) |
| C7 | 프로젝트 이름을 바꾸고 싶음(폴더명 변경) | ⚠️ | v1 미지원 — 수동 폴더 rename 시 wsId·선언 정합 절차 미정의. 백로그 후보 |
| C8 | 역할을 제거하고 싶음 | ⚠️ | 역할 추가(FS-8 병합)는 있으나 **제거 경로 없음** — project.json 수동 편집이 현재 유일. [＋ 역할] 결정과 함께 다룰 후보 |
| C9 | 대시보드 탭을 여러 개 열어둠 | ◐ | 폴링 중복은 무해 — 양쪽에서 동시 액션 시 멱등이 방어(활성화)하나 비활성화 이중 확인은 한쪽 404/무해 실패 예상 — 확인만 |
| C10 | 세션 행을 눌렀는데 wmux 창이 최소화되어 있음 | ◐ | /attach가 포커스는 옮기나 창 복원(foreground)까지 되는지 wmux 실측 필요 |
