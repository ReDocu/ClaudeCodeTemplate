## 2. 실행 우선순위 — 확신도 보정 RICE

## 3. 트랙 분리 — Build vs Validate (핵심 종합)

`03`과 `04`의 충돌을 푸는 열쇠: **모든 항목을 같은 로드맵에 줄 세우지 않는다.** 확신도와 노력으로 4분면에 나눈다.

```
           낮은 Effort                     높은 Effort
        ┌───────────────────────┬───────────────────────┐
 높은   │  🟢 T0 지금 만든다      │  🟡 T1 다음에 만든다     │
 확신   │  버튼시맨틱·포커스트랩   │  이중부호화·accept루프   │
        │  라이브리전·저대비      │  온보딩빈상태            │
        │  카운트내비·종료안전    │                        │
        ├───────────────────────┼───────────────────────┤
 낮은   │  🔵 T3 기회 되면         │  🔴 T2 먼저 검증한다     │
 확신   │  보안배지·MCP·오류복구   │  라이브니스(E1)          │
        │                        │  드로어승인(E2)·청구(E3) │
        └───────────────────────┴───────────────────────┘
```

- **🟢 T0 (지금 만든다)**: 고확신·저비용. 게이트 불필요. 안전·접근성 정확성·값싼 트리아지.
- **🟡 T1 (다음)**: 확신 양호, 노력 중간. T0 직후 착수.
- **🔴 T2 (먼저 검증)**: **`03`의 간판 베팅 3종이 여기 있다.** 잠재력은 크나 확신이 낮고 비싸다 → **실험 통과 전 엔지니어링 금지.**
- **🔵 T3 (기회 되면)**: 저확신·저가치. 여유 있을 때.

## 4. 실행 시퀀스 (2트랙 병행)

### 트랙 A — Build (바로 손대는 것)
**Now (T0)** — 안전·접근성 차단·값싼 트리아지
1. 파괴적 종료 안전화: 확인 + undo + 위험 액션 시각 분리 (F5)
2. 키보드 포커스 트랩 제거 + 인박스 버튼 시맨틱 (F1·F10)
3. 파괴적/대기 전이 라이브 리전 호명 (F11·F12)
4. 카운트→세션 내비게이션 + 저대비 텍스트 상향 (F2·F13)

**Next (T1)** — 트리아지·행동·명료함 확장
5. 포커스 이동/복귀 (F9) · 신호등 최악-자식 롤업 (F3)
6. 상태 색+형태+텍스트 이중부호화 + 범례 형태 키 (F14·F15)
7. 인수인계 accept/kick-back 루프 (F6)
8. 온보딩 빈상태 + 복사 명령, CLI/대시보드 분리 교육 (F23·F24)

### 트랙 B — Validate (병렬로 지금 돌리는 실험, 결과가 게이트)
| 실험 | 검증 대상 베팅 | 통과 기준 → 승격 | 실패 시 |
|------|---------------|------------------|---------|
| **E1** stale 라벨 A/B | 라이브니스(F4) | 화면 기반 종료 실행률 유의 상승 | 위생 항목으로 강등 |
| **E2** 드로어 승인 버튼 프로토타입 | 인라인 승인(F8) | 세션 진입 ↓15%↑ **且** 오승인율 무상승 | 위험 액션은 제외 |
| **E3** 롤업 목업 + 실제 가격 노출 | 청구(M4) | 결제 의도 클릭 유의미 | 수익화 축 교체 |

> **왜 병행인가**: 트랙 B는 코드가 아니라 프로토타입/목업이라 **트랙 A와 자원 경합이 적다.** 레드팀 A6("순서가 사업적으로 역방향")의 처방 = 비싼 베팅을 뒤로 미루되 **검증은 지금 당장** 시작.

## 6. 프레임워크 주의 & 한계 (정직 표기)

- **데이터 한계**: n=8 정성 + 쿼터 표본. 모든 Reach·Confidence는 **모집단 추정이 아니다.** RICE 절대값은 신뢰하지 말고 **상대 순위와 트랙 구분**만 사용.
- **Effort는 추정**: 엔지니어링 리뷰 전 값. E가 바뀌면 7~11위권 순서는 흔들릴 수 있음(T0/T2 경계는 견고).
- **Opportunity Score의 맹점**: 응답이 전원 부정이라 Satisfaction이 상수처럼 작동 → 사실상 Importance만 반영. 그래서 **Reach·Confidence를 RICE에서 별도 반영**했다.
- **솔루션 편향 잔존**: CSV의 `insight` 열이 이미 솔루션이었다. 2절 항목명은 솔루션이지만, **1절에서 문제(O1~O10)로 되돌려** 점수화했다. 로드맵 실행 시 솔루션은 A/B로 열어둘 것.
- **다음 관문**: T0 배포 후 **정량 라운드**(트리아지 속도·오클릭율·과업 완료율)로 이 순위를 재검증. 그전까지 `03`의 "확신도 높음"은 전부 잠정.

## 7. 한 문단 요약 (경영진용)

네 문서를 종합한 최종 우선순위의 결론은 **2트랙**이다. **지금 만들 것(T0)** 은 값싸고 확실한 것 — 파괴적 종료 안전화, 접근성 치명 블로커(포커스 트랩·인박스 버튼·라이브 리전), 값싼 트리아지 내비게이션 — 으로, RICE 상위를 독식한다. 반면 `03`이 간판으로 내세운 세 베팅(**라이브니스·드로어 인라인 승인·청구 롤업**)은 확신도가 낮고 비싸 **RICE 하위(0.1~0.4)**로 내려앉으며, 코드가 아니라 **실험(E1·E2·E3)으로 먼저 검증**한 뒤에만 승격한다. 프로토타입 기반이라 두 트랙은 **병행 가능**하므로, 레드팀이 지적한 "매출 신호를 맨 뒤로 미루는 역방향 순서"를 피한다. 마지막으로 `03`의 **"잠재시장 37.5%" 문구는 즉시 정정**해야 한다. 모든 수치는 n=8 정성 기반이므로 **상대 순위·트랙 구분에만** 쓰고, T0 배포 후 정량 라운드로 재검증한다.
